# AGENTS.md - Woolang Codebase Guide

## Project Overview

Woolang is a statically-typed scripting language with three codebases:
- **`src/` + `driver/`** ŌĆö C++17 compiler, runtime linker, and public C API (`include/wo.h`)
- **`3rd/woort/`** ŌĆö C11 VM runtime (git submodule, has its own `AGENTS.md`)
- **`woo/`** ŌĆö Woolang standard library (`.wo` files, **embedded into the binary at build time**)

## Prerequisites

```bash
git submodule update --init --recursive --force
```

Submodule `3rd/woort` is required before any build. Woort itself has a sub-submodule `3rd/woomem` (GC allocator).

## Build

### Windows (MSVC)
```bash
cmake -B build -A x64 -DWO_MAKE_OUTPUT_IN_SAME_PATH=ON -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=ON
cmake --build build --config Release --parallel
```

### Linux / macOS
```bash
cmake -B build -DWO_MAKE_OUTPUT_IN_SAME_PATH=ON -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=ON
cmake --build build -j$(nproc)
```

Without `WO_MAKE_OUTPUT_IN_SAME_PATH=ON`, binaries go into platform-specific build subdirectories.

### Build outputs

| Target | CMake name | Output |
|---|---|---|
| Compiler library | `woolang` | `libwoo.dll` / `libwoo.so` / `libwoo.dylib` |
| Executable | `woodriver` | `woolang.exe` / `woolang` |
| Runtime | `woort` | `libwoort.dll` / `libwoort.so` / `libwoort.dylib` |

Debug builds append `_debug` suffix (64-bit) or `32_debug` (32-bit): `woolang_debug`, `libwoo_debug`, `libwoort_debug`.

### Key CMake options

- `BUILD_SHARED_LIBS=ON` (default) ŌĆö shared library output
- `WO_DISABLE_COMPILER=ON` ŌĆö runtime-only, no compiler frontend
- `WO_BUILD_FOR_COVERAGE_TEST=ON` / `WO_BUILD_FOR_ASAN_TEST=ON` ŌĆö CI instrumentation
- `WO_DISABLE_FUNCTION_FOR_WASM=ON` ŌĆö strip dynamic lib loading for WASM builds

## Auto-generated files (do not edit)

- `src/wo_commit_sha.hpp` ŌĆö current git hash, generated by CMake
- `src/wo_lang_grammar_lr1_autogen.hpp` ŌĆö LR(1) parser tables, generated externally
- `<build>/gen/wo_stdlib_embedded.inc` ŌĆö embedded woolang stdlib from `woo/*.wo`, generated by CMake

Adding or removing `.cpp`/`.hpp` files in `src/` requires no CMakeLists changes (uses `GLOB_RECURSE`).

## Testing

Tests are written **in Woolang** (`.wo` files under `test/`), run by the built `woolang` binary. Before running tests, install test dependencies:

```bash
cd test && baozi install && cd ..
```

Run all tests:
```bash
./build/woolang ./test/test_all.wo --woort-gc-max-reserved-memory 2560
```

Run a single test module:
```bash
./build/woolang ./test/test_basic.wo --woort-gc-max-reserved-memory 2560
```

Compile-then-run test (catches compiler errors separately):
```bash
./build/woolang ./test/test_all.wo -o test.woo && ./build/woolang ./test.woo --woort-gc-max-reserved-memory 2560
```

Test framework helpers: `test_function()`, `test_equal()`, `test_assure()` (defined in `test/test_tool/test_tool.wo`).

The `test/package.json` is for the `baozi` package manager only ŌĆö there is no Node.js toolchain.

## CI

**GitLab CI only** (`.gitlab-ci.yml`). No GitHub Actions. Jobs build and test on Windows MSVC, Ubuntu 20.04 x86_64, Ubuntu 22.04 ARM64, and macOS ARM64.

## Code style (C++ ŌĆö `src/` and `driver/`)

- C++17, no exceptions (`-fno-exceptions` on non-MSVC)
- Precompiled header: `src/wo_afx.hpp`
- MSVC: `/source-charset:utf-8`, `/D_CRT_SECURE_NO_WARNINGS`
- Public C API header: `include/wo.h`

## Code style (Woolang runtime ŌĆö `3rd/woort/`)

Read `3rd/woort/AGENTS.md` for full conventions. Key points:
- C11 only, `/* */` comments only, `#pragma once`
- `WOORT_NODISCARD` required on every non-void function
- `/* OPTIONAL */` annotation required before any nullable type
- Naming: `woort_ModuleName` for types, `woort_module_function` for functions

## No lint / formatter / typecheck

There are no lint, formatter, or typecheck commands in this repo. Do not attempt to run them.

## Architecture

The compiler pipeline (in `src/`):
1. **Lexer** (`wo_compiler_lexer.cpp`) ŌĆö tokenization + macro handling
2. **Parser** (`wo_compiler_parser.cpp`) ŌĆö LR(1) driven by auto-generated grammar
3. **Pass 0** (`wo_lang_pass0.cpp`) ŌĆö scope resolution, symbol registration
4. **Pass 1** (`wo_lang_pass1.cpp`) ŌĆö type checking, type deduction, constant evaluation
5. **Pass IR** (`wo_lang_passir.cpp`) ŌĆö bytecode generation via IR compiler
6. **Woort VM** (submodule) ŌĆö executes generated bytecode with GC

The standalone executable entrypoint is `driver/wo_driver.cpp`.
